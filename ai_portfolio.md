## 关于我

我长期在飞书 / Lark 从事商业化策略、经营分析、数据产品相关工作。过去几年，我一边在真实业务环境里处理复杂问题，一边持续尝试用 AI Agent、workflow 和自动化工具解决具体问题。

我对 AI 的兴趣不只是“会不会用模型”，而是更关心三个问题：

- AI 是否能真正承接一部分工作，而不只是回答问题
- Agent 如何在真实场景里与人协作，而不是脱离流程单独存在
- 一个工具是否能足够简单、稳定、有价值，值得被持续使用

下面这些项目，大多不是为了展示技术炫技，而是来自真实问题、真实流程和真实使用需求。

## 项目一：VOC Agents

**一句话介绍**  
一个自动从 Lark 群聊中拉取用户产品反馈、进行结构化整理的 Agent。

**为什么做**  
在产品和业务协作场景里，用户反馈经常分散在聊天记录中。人工收集的问题不是“看不到”，而是很难持续、完整、低成本地整理，导致高价值反馈容易流失，后续分析和流转效率也低。

**我做了什么**

- 从真实反馈收集场景出发，设计一个开箱即用的 Agent，用于自动拉取 Lark 群聊中的用户反馈。
- 设计 Agent 的 onboarding 方式，让使用者能够更低门槛地接入实际工作流。
- 参与和思考 skill 设计、任务拆解、人机协作边界，以及结构化输出方式。
- 把重点放在“如何真的被用起来”，而不是只完成一次性的 demo。

**项目中的核心思考**

- Agent 不能只会“看消息”，还需要理解反馈收集在整个产品流程中的位置。
- 一个有价值的 Agent，不只是自动化提取信息，还要让后续的人能够接住、判断和继续推进。
- 人机协作流程设计和 onboarding 体验，很多时候比单点模型效果更决定是否落地。

**这个项目体现的能力**

- 从真实问题出发定义产品形态
- 把 workflow 拆解成 Agent 可承接的任务
- 设计 skill 与人机协作机制
- 平衡“可做出来”和“可被持续使用”

## 项目二：AI Tri Coach

**一句话介绍**  
一个面向铁人三项训练场景的 AI 教练 Agent。

**为什么做**  
个人训练过程中，经常会遇到训练安排、身体反馈、伤病与恢复建议分散、难以连续追踪的问题。通用聊天工具可以提供建议，但很难结合长期训练历史做出更有针对性的支持。

**我做了什么**

- 设计并推进一个 AI 驱动的铁人三项教练 Agent，围绕训练建议、任务编排和个性化支持进行探索。
- 在项目中重点关注 Agent runtime、memory、workflow 等核心机制如何服务真实长期场景。
- 思考如何让 Agent 利用历史上下文理解用户，而不是每次从零开始响应。

**项目中的核心思考**

- memory 的价值不只是“记住信息”，而是让 Agent 能基于过去经历形成连续理解。
- 对长期任务来说，Agent 更像一个持续协作者，而不是一次性问答工具。
- 真正有价值的 Agent 需要把建议、上下文和后续行动连接起来。

**这个项目体现的能力**

- 能从个人真实痛点中抽象产品方向
- 对 Agent runtime、memory、workflow 有实际拆解和思考
- 能把模糊想法快速转成原型与验证路径

## 项目三：问数 Agent

**一句话介绍**  
一个面向日常数据分析工作的自用问数 Agent。

**为什么做**  
在经营分析和商业化分析工作中，大量时间消耗在重复查询、口径确认、问题澄清和信息往返上。很多问题并不复杂，但响应链路长，影响工作效率。

**我做了什么**

- 结合自己的日常分析场景，搭建自用问数 Agent，用于提升查询、分析和信息获取效率。
- 探索如何把常见分析问题转成更自然的交互方式，减少重复性工作。
- 在使用过程中验证什么样的问法、流程和输出结构更适合真实工作。

**项目中的核心思考**

- 数据分析类 Agent 的关键不只是“回答一个数”，而是减少理解问题和组织答案的摩擦。
- 工具是否高频可用，往往取决于交互成本和输出是否能直接接入后续工作。
- 自用工具是很好的试验场，能快速暴露真正的效率瓶颈。

**这个项目体现的能力**

- 能把业务流程中的低效点转成可验证的工具形态
- 对分析工作流和用户交互有产品感
- 重视工具的实用性而不只是功能堆叠

## 项目四：高价值客户归属扫描 Agent

**一句话介绍**  
一个帮助识别和处理高价值客户归属问题的 Agent。

**为什么做**  
商业化分析工作里，经常会遇到客户归属相关的识别和核查问题。这类问题对业务判断重要，但人工处理成本高、过程繁琐，也容易遗漏。

**我做了什么**

- 搭建高价值客户归属扫描 Agent，用于支持归属相关问题的识别和处理。
- 将具体业务问题转化为可执行的 Agent 流程，帮助提高分析与协同效率。
- 在项目中持续思考 Agent 在业务判断类场景中的边界，避免工具输出与人工判断脱节。

**项目中的核心思考**

- 在业务场景里，Agent 不一定替代判断，但可以显著提升信息筛查和前置处理效率。
- 对这类场景来说，好的 Agent 应该帮助人更快地看见问题、缩小范围、提高决策质量。
- 业务类 Agent 需要更强的流程意识，而不是单点智能。

**这个项目体现的能力**

- 能将抽象业务问题结构化
- 能把复杂判断前的准备工作交给 Agent
- 理解 Agent 在业务系统中的角色和边界

## 我如何做 AI 项目

我的做法通常不是先想“模型能做什么”，而是先想：

1. 真实工作或真实生活里，哪个环节最值得被重做
2. 这个问题适不适合让 Agent 承接，而不是只做一个聊天界面
3. 工作流里哪些步骤应该自动化，哪些步骤应该保留给人判断
4. 什么样的输出能真正进入后续流程，而不是停留在 demo

我会特别关注以下几个方面：

- 问题是否真实存在，而不是伪需求
- onboarding 是否足够简单，别人是否容易用起来
- 输出是否结构化，能否衔接后续动作
- 人机边界是否清晰，是否有合理的协作方式
- 工具是否真的提升效率，而不只是“看起来像 AI”
