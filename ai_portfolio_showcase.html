AI Portfolio / Builder Case Studies

探索人与 Agent 协作

郭择良

我做 AI 项目时，最关心的不是模型能不能回答问题，而是人和 Agent 如何在真实场景里分工、接力、形成长期协作。

无论是面向个人训练与伤病恢复的 TriCoach，还是面向组织反馈流转的 VOC Agents，我想探索的其实是同一个问题：哪些事情适合交给 Agent 承接， 哪些判断需要由人保留，以及两者如何围绕同一份上下文持续配合。

我希望做的，不只是更聪明的 AI，而是真正能与人协作、沉淀上下文并持续创造价值的产品。

项目一：TriCoach

一个连接 Strava 的 AI 铁三教练系统。它不是简单分析运动数据，而是把训练同步、 体能指标、对话复盘、教练人设和长期记忆连接成一个持续协作体验。

Strava Integration
Training Analytics
Proactive Coaching
Conversation Memory
Why It Matters

我关注的不是“能聊”，而是“能长期陪练”

从 repo 可以看到，这个项目已经形成了比较完整的产品闭环：Strava 数据同步、 CTL/ATL/TSB 体能指标、训练日志、赛季规划、比赛前后提醒、以及教练对话页面。
  • 后端基于 FastAPI，前端是 Next.js，已经不是单页 demo，而是完整 web 产品结构。
  • 核心能力不是一次性问答，而是围绕训练复盘、疲劳预警、比赛准备、长期日志积累的持续服务。
  • 设计文档里明确区分了 THINK / CHAT / EXTRACT 三档任务，并给不同模型分配不同职责。
  • 对话系统会结合近 7 天训练记录、阈值数据和当前体能状态生成教练 prompt，而不是通用聊天。
  • 我特别关注 Memory 的产品价值：对话不是即时消耗，而是会沉淀为训练笔记、周总结、月总结和赛季记录，供后续多层调用。
  • 在伤病和恢复场景下，我关注的不只是给建议，而是设计一个带 loop 的 agent runtime：识别异常状态，持续追问上下文，记录恢复进展，并在后续对话中继续接住这个问题。
这个项目最能体现我的一点是：我在意 Agent 如何把一次对话变成长期记忆，并在真实生活场景里持续工作，而不是只做一轮漂亮回复。

Design Breakdown
1. 数据接入

通过 Strava OAuth 接入训练数据，用户不用手动维护训练记录。

2. 状态建模

用 CTL / ATL / TSB、阈值和近 7 天训练作为教练判断上下文。

3. 对话复盘

每次训练后主动收集训练类型、RPE、身体状态和生活干扰，不把交互做成生硬填表。

4. 长期记忆

把对话沉淀成训练笔记、周总结、月总结和赛季总结，让教练具备可被后续多层调用的连续理解。

项目二：VOC Agents

一个从飞书群聊自动采集用户反馈、经 AI 清洗后写入 Lark Base 的多 Agent 系统。 它最有价值的地方，不是“识别反馈”本身，而是把采集、清洗、分析、报告做成了一条可重复运行的工作流。

Multi-Agent

Why It Matters

不是做一个 Agent，而是做一个能接入组织流程的骨架

这个 repo 的亮点非常清楚：它不是单功能脚本，而是一个以 VOC 为首个落地场景的通用 Agent 协作平台。 真正的产品感体现在 CLI 初始化、角色注入、状态追踪和 Lark Base 数据骨架上。
  • `voc_agents init` 不是简单建表，而是带着用户完成群扫描、Base 初始化、角色配置和首轮运行。
  • 设计文档把系统拆成 Project、Agent、Human Master、Task、Skill、Context，边界很清晰。
  • 核心原则是“共享事实层 + 私有分析层”，避免分析结果反向污染底层事实数据。
  • 通过 Lark Base 作为数据总线，让 Collector / Cleaner / Analyst 等不同角色协同工作。
这个项目最能体现我的一点是：我不只想做一个会跑的 agent，而是想把 agent 放进真实协作流程里，让它和人一起工作。

Project Lark 群（通信总线）
├── 任务发布   → Human 在群里发消息 @ Agent
├── Agent 接收 → OpenClaw 监听群 chat_id，@ 触发对应 skill
├── 任务结果   → Agent 写 Lark Base + 群里广播
└── 关键事件   → Platform Bot 广播到群

Lark Base（数据与配置中心）
├── 配置层：project_config / agent_registry / skill_registry / init_checklist
├── 任务层：tasks / task_logs
├── 数据层：raw_messages / raw_voices / cleaned_voices
└── 产出层：demands / outputs

OpenClaw Agent（每个 Agent 独立 Lark 身份）
├── channel:       project_group_chat_id
├── trigger:       @ pattern（@Cleaner、@Analyst...）
└── system_prompt: 通用 skill + 角色 skill（初始化时 human_master 指令注入）
System Architecture 从群聊触发、到 Base 写入、再到角色分工，构成一条完整协作链路。
cleaned_voices Fact Layer 原始消息经过清洗后进入共享事实层，再供后续分析、聚合和报告使用。
Design Breakdown
1. 项目初始化

一条命令完成环境检查、反馈群选择、Base 创建和角色数据写入。

2. 角色协作

Collector 拉消息，Cleaner 清洗为结构化事实，Analyst 基于事实层生成报告和洞察。

3. 事实层设计

以 cleaned_voices 作为共享公共事实，后续分析建立在稳定底层之上。

4. 产品化思路

把首轮可用体验拆成“初始化项目”和“拿到第一份报告”两段，减少首次使用负担。

我在这些项目里体现出的能力

这两个项目表面上一个偏个人应用，一个偏组织工作流，但它们背后其实是同一套方法： 找真实问题，拆成能被 Agent 承接的步骤，再设计人机边界和可持续使用方式。

从模糊问题出发

我不是从已有方案里选功能，而是先看真实流程里最值得被重做的那一段，再反推产品形态。

把想法拆成工作流

我更关注上下游如何接住，而不是只做中间那一段智能。好的 Agent 必须接得上真实流程。

重视 onboarding

项目能不能被别人真的用起来，往往取决于接入成本、默认路径和第一次成功体验。

理解 memory 的价值

记忆不是多存一点上下文，而是把对话沉淀为后续可调用的结构化经验，服务长期任务而不是单轮问答。

用结构稳定系统

在 VOC Agents 里，我会优先稳住事实层和角色边界，因为没有稳定结构，Agent 协作很快会失真。

把 AI 做成真实工具

我在意的不是“这个东西像不像 AI”，而是“它有没有把某项工作真的变得更简单、更快、更可持续”。
